## 1. 项目背景与目标

### 1.1 背景

企业内部已有多个 AI 智能体（如 OpenClaw Agent、Claude Code、自研 Agent 等），它们需要与企业微信（WeCom）进行交互：

- 管理员工日程（创建/查询/修改/删除日程）
- 预订和管理会议室
- 向员工或群组发送应用消息（文本、图片、文件、卡片等）

当前各智能体各自直连企业微信 API 存在以下问题：

- **密钥散落**：每个智能体都需要持有 CorpID、AgentID、Secret 等敏感凭证，管理成本高、泄露风险大
- **能力重复建设**：每个智能体都需要自行实现企业微信 API 调用、token 管理、错误处理等
- **缺乏管控**：无法统一审计谁在什么时间执行了什么操作，无法做权限控制
- **接入门槛高**：新智能体接入需要理解企业微信 API 细节，开发效率低

### 1.2 目标

构建 **WeCom Gateway**，作为所有 AI 智能体与企业微信交互的统一入口：

- **统一 API**：提供简洁的 RESTful API，智能体只需调用 HTTP 接口即可完成所有企业微信操作
- **凭证托管**：所有企业微信密钥由网关统一管理，智能体不接触任何密钥
- **鉴权体系**：智能体通过 API Key 认证身份，不同智能体拥有不同权限
- **审计日志**：所有操作完整记录，支持查询和追溯
- **多租户支持**：支持多个企业微信应用（不同 AgentID/Secret），统一管理

### 1.3 非目标

- 不实现智能体本身的业务逻辑
- 不实现企业微信的所有 API（仅覆盖日程、会议室、消息发送等核心场景）
- 不实现前端管理界面（MVP 阶段通过 API 和配置文件管理）
- 不实现消息接收（仅负责主动发送和资源管理）

## 2. 用户角色

| 角色 | 描述 |
|------|------|
| **AI 智能体** | API 调用者，通过 API Key 认证后调用网关提供的接口 |
| **企业管理员** | 系统管理员，负责配置企业微信凭证、管理 API Key、查看审计日志 |
| **最终用户** | 企业微信中的员工，日程/会议室/消息的目标对象（不直接使用本系统） |

## 3. 功能需求

### 3.1 鉴权与访问控制

#### FR-001：API Key 认证

**优先级**：P0

**描述**：所有 API 请求必须通过 API Key 认证。

**验收标准**：
- 每个智能体分配唯一的 API Key（格式：`wgk_<random_32chars>`）
- API Key 通过 `Authorization: Bearer wgk_xxxx` 请求头传递
- 无效或缺失 API Key 的请求返回 `401 Unauthorized`
- API Key 支持启用/禁用状态，禁用的 Key 返回 `403 Forbidden`
- API Key 支持过期时间，过期的 Key 返回 `401 Unauthorized`

#### FR-002：API Key 权限控制

**优先级**：P0

**描述**：每个 API Key 绑定一组权限，控制可访问的 API 范围。

**验收标准**：
- 权限以能力（capability）为单位，包括：`calendar:read`、`calendar:write`、`meetingroom:read`、`meetingroom:write`、`message:send`
- 每个API Key 可配置一个或多个权限
- 请求超出权限范围时返回 `403 Forbidden`
- 支持通配符权限 `*` 表示全部权限（仅限管理员 Key）

#### FR-003：API Key 管理

**优先级**：P0

**描述**：管理员可创建、查询、禁用、删除 API Key。

**验收标准**：
- 提供 RESTful API 进行 Key 的 CRUD 操作
- 创建 Key 时需指定名称、权限列表、过期时间（可选）
- 创建时返回完整 Key（仅此一次可见），后续查询仅返回 Key 前缀（`wgk_abc...`）
- 支持 Key 列表查询（分页）

### 3.2 企业微信日程管理

#### FR-010：创建日程

**优先级**：P0

**描述**：为指定员工创建企业微信日程。

**请求参数**：
- `organizer`（必填）：组织者 UserID
- `summary`（必填）：日程主题
- `description`（可选）：日程描述
- `start_time`（必填）：开始时间（RFC3339）
- `end_time`（必填）：结束时间（RFC3339）
- `attendees`（可选）：参与者 UserID 列表
- `location`（可选）：地点
- `remind_before_minutes`（可选）：提前提醒分钟数，默认 15

**验收标准**：
- 调用企业微信 `schedule` API 创建日程
- 成功返回日程 ID 和详细信息
- 参数校验：开始时间必须早于结束时间
- 企业微信 API 错误正确映射为 HTTP 错误码

#### FR-011：查询日程

**优先级**：P0

**描述**：查询指定员工的日程列表。

**请求参数**：
- `userid`（必填）：目标员工 UserID
- `start_time`（可选）：筛选起始时间
- `end_time`（可选）：筛选结束时间
- `cursor`（可选）：分页游标
- `limit`（可选）：每页数量，默认 50，最大 100

**验收标准**：
- 返回日程列表，包含日程 ID、主题、时间、参与者等信息
- 支持时间范围筛选
- 支持游标分页

#### FR-012：更新日程

**优先级**：P1

**描述**：更新已有日程的信息。

**验收标准**：
- 支持部分更新（只传需要修改的字段）
- 组织者、主题、时间、参与者、地点等均可修改
- 日程不存在时返回 `404 Not Found`
- 仅日程组织者或管理员可更新

#### FR-013：取消日程

**优先级**：P1

**描述**：取消（删除）已有日程。

**验收标准**：
- 通过日程 ID 取消日程
- 日程不存在时返回 `404 Not Found`
- 仅日程组织者或管理员可取消

### 3.3 会议室管理

#### FR-020：查询会议室列表

**优先级**：P0

**描述**：查询企业可用的会议室列表。

**请求参数**：
- `city`（可选）：按城市筛选
- `building`（可选）：按楼宇筛选
- `floor`（可选）：按楼层筛选
- `capacity`（可选）：最小容量
- `equipment`（可选）：设备筛选（如"投影仪"、"视频会议"）
- `cursor`（可选）：分页游标
- `limit`（可选）：每页数量

**验收标准**：
- 返回会议室列表，包含 ID、名称、容量、位置、设备等信息
- 支持多条件组合筛选
- 支持分页

#### FR-021：查询会议室空闲时段

**优先级**：P0

**描述**：查询指定会议室在给定时间范围内的空闲时段。

**请求参数**：
- `meetingroom_id`（必填）：会议室 ID
- `start_time`（必填）：查询起始时间
- `end_time`（必填）：查询结束时间

**验收标准**：
- 返回该会议室在指定时间段内的空闲时段列表
- 时段以 `[start, end)` 半开区间表示
- 与已有日程不冲突的时段才标记为空闲

#### FR-022：预订会议室

**优先级**：P0

**描述**：预订会议室并自动创建关联日程。

**请求参数**：
- `meetingroom_id`（必填）：会议室 ID
- `subject`（必填）：会议主题
- `start_time`（必填）：开始时间
- `end_time`（必填）：结束时间
- `booker`（必填）：预订人 UserID
- `attendees`（可选）：参会者 UserID 列表

**验收标准**：
- 调用企业微信会议室预订 API
- 预订成功返回预订 ID 和日程 ID
- 时间冲突时返回 `409 Conflict` 并提示冲突的已有预订
- 开始时间不得早于当前时间

### 3.4 应用消息发送

#### FR-030：发送文本消息

**优先级**：P0

**描述**：向指定用户或部门发送文本消息。

**请求参数**：
- `receiver_type`（必填）：接收者类型，`user` / `department` / `tag`
- `receiver_ids`（必填）：接收者 ID 列表
- `content`（必填）：文本内容
- `safe`（可选）：是否为保密消息，默认 false

**验收标准**：
- 通过企业微信应用消息 API 发送
- 成功返回消息发送结果
- 单次最多支持 1000 个接收者

#### FR-031：发送 Markdown 消息

**优先级**：P0

**描述**：向指定用户或部门发送 Markdown 格式消息。

**验收标准**：
- 支持 Markdown 格式文本
- 企业微信支持的 Markdown 子集均可使用

#### FR-032：发送图片消息

**优先级**：P1

**描述**：向指定用户或部门发送图片消息。

**请求参数**：
- `receiver_type`、`receiver_ids`：同文本消息
- `image_url`（必填）：图片 URL（企业微信会自动下载）
- 或 `media_id`（二选一）：已上传的素材 ID

**验收标准**：
- 支持通过 URL 或已上传素材发送图片
- 图片格式支持 JPG、PNG
- 图片大小不超过 2MB（企业微信限制）

#### FR-033：发送文件消息

**优先级**：P1

**描述**：向指定用户或部门发送文件消息。

**验收标准**：
- 支持通过 URL 发送文件（网关先上传素材再发送）
- 文件大小不超过 20MB（企业微信限制）

#### FR-034：发送卡片消息

**优先级**：P1

**描述**：向指定用户或部门发送交互式卡片消息。

**请求参数**：
- `receiver_type`、`receiver_ids`：同文本消息
- `card_content`（必填）：卡片 JSON 内容（符合企业微信模板卡片格式）

**验收标准**：
- 支持文本通知型卡片和按钮交互型卡片
- 卡片内容校验（基本结构完整性检查，不校验业务语义）

### 3.5 审计日志

#### FR-040：操作审计

**优先级**：P0

**描述**：所有 API 操作自动记录审计日志。

**验收标准**：
- 记录字段：时间、API Key 名称、请求方法、请求路径、请求参数（脱敏）、响应状态码、响应耗时、客户端 IP
- 密码、Secret、API Key 等敏感字段在日志中脱敏（`***` 替换）
- 日志持久化存储，不随服务重启丢失

#### FR-041：审计日志查询

**优先级**：P1

**描述**：管理员可查询审计日志。

**请求参数**：
- `api_key_name`（可选）：按 API Key 筛选
- `method`（可选）：按 HTTP 方法筛选
- `path`（可选）：按请求路径筛选
- `status_code`（可选）：按响应状态码筛选
- `start_time` / `end_time`（可选）：按时间范围筛选
- `cursor` / `limit`：分页

**验收标准**：
- 支持多条件组合筛选
- 按时间倒序排列
- 支持分页

### 3.6 系统管理

#### FR-050：健康检查

**优先级**：P0

**验收标准**：
- `GET /health` 返回服务健康状态
- 检查项：数据库连接、企业微信 API 可达性
- 响应格式：`{"status": "ok", "version": "x.y.z", "checks": {...}}`

#### FR-051：企业微信凭证管理

**优先级**：P0

**描述**：管理员可配置和管理企业微信应用凭证。

**验收标准**：
- 支持配置多个企业微信应用（CorpID + AgentID + Secret）
- 凭证加密存储（AES-256-GCM）
- 提供管理 API 进行凭证的 CRUD
- 凭证变更后自动刷新 access_token

### 3.7 多企业支持

#### FR-060：多 CorpID 管理

**优先级**：P1

**描述**：支持同时管理多个企业微信企业，不同企业之间资源隔离。

**验收标准**：
- 支持配置多个 CorpID 及其下属应用
- API Key 绑定到具体的 CorpID + App 组合
- 不同 CorpID 的 access_token 独立管理
- 企业微信 API 调用时自动使用对应 CorpID 的凭证
- 管理员可按 CorpID 查询应用和 API Key

### 3.8 gRPC 接口

#### FR-070：gRPC 服务端

**优先级**：P1

**描述**：除 REST API 外，同时暴露 gRPC 接口，供对性能和类型安全有要求的智能体使用。

**验收标准**：
- 使用 Protobuf 定义与 REST API 等价的服务接口
- gRPC 端口与 HTTP 端口可独立配置
- 认证方式：gRPC metadata 传递 API Key（`authorization: Bearer wgk_xxxx`）
- 支持流式接口（stream）：日程变更通知、消息发送进度等
- 错误码映射：gRPC status code ↔ HTTP status code

#### FR-071：gRPC 拦截器

**优先级**：P1

**验收标准**：
- 认证拦截器：验证 gRPC metadata 中的 API Key
- 审计拦截器：记录 gRPC 调用日志（与 HTTP 审计统一格式）
- 限流拦截器：复用 HTTP 的令牌桶限流逻辑
- Recovery 拦截器：捕获 panic 防止服务崩溃

### 3.9 API 文档

#### FR-080：OpenAPI 规范生成

**优先级**：P0

**描述**：自动生成 OpenAPI 3.0 规范文档，供智能体开发者参考。

**验收标准**：
- 通过代码注释（swagger annotations）自动生成 OpenAPI spec
- 端点 `GET /v1/openapi.json` 返回完整的 OpenAPI JSON spec
- 端点 `GET /v1/docs` 提供内置的 Swagger UI 页面
- 文档包含所有 API 端点、请求参数、响应格式、错误码、认证方式
- 支持在线调试（Try it out）

### 3.10 管理后台

#### FR-090：管理后台 Web UI

**优先级**：P1

**描述**：提供 Web 管理界面，管理员可通过浏览器完成所有管理操作，无需调用 API。

**验收标准**：

**API Key 管理**：
- 查看 API Key 列表（名称、权限、状态、过期时间、创建时间）
- 创建 API Key（指定名称、权限、过期时间）
- 禁用/启用 API Key
- 删除 API Key

**企业微信应用管理**：
- 查看应用列表（CorpID、AgentID、状态）
- 添加/编辑/删除应用
- 查看 access_token 状态和过期时间
- 手动刷新 access_token

**审计日志查看**：
- 按时间、API Key、路径、状态码筛选日志
- 分页浏览
- 导出日志（CSV 格式）

**仪表盘**：
- 总请求数、成功率、错误率统计
- 各 API Key 调用量排行
- 最近 24h 请求趋势图

## 4. 非功能需求

### NFR-001：性能

- API 响应时间 P99 < 2s（不含企业微信 API 耗时）
- 支持至少 100 QPS 并发
- 企业微信 access_token 缓存，避免频繁获取

### NFR-002：可靠性

- 数据库操作失败时正确回滚
- 企业微信 API 调用失败时返回有意义的错误信息
- 服务优雅关闭，不丢失进行中的请求

### NFR-003：安全性

- API Key 使用 Argon2id 或 bcrypt 哈希存储
- 企业微信凭证使用 AES-256-GCM 加密存储
- 审计日志不可篡改（append-only）
- 请求频率限制（rate limiting），防止滥用
- HTTPS 强制（生产环境）

### NFR-004：可观测性

- 结构化日志（JSON 格式）
- 关键指标暴露（Prometheus 格式）：请求量、延迟、错误率、Token 刷新次数
- 健康检查端点

### NFR-005：部署

- 单二进制部署
- 配置通过环境变量 + 配置文件（YAML）
- 数据库支持 SQLite（轻量部署）和 PostgreSQL（生产部署）

## 5. 技术约束

| 约束 | 说明 |
|------|------|
| 开发语言 | Golang 1.23+ |
| HTTP 框架 | Gin |
| gRPC 框架 | google.golang.org/grpc + grpc-gateway（HTTP↔gRPC 双协议） |
| API 文档 | swaggo/swag（注释生成 OpenAPI 3.0 spec） |
| 数据库 | SQLite（开发）/ PostgreSQL（生产），通过接口抽象 |
| 前端 | 管理后台使用轻量级 SPA（嵌入式静态资源，单二进制分发） |
| 企业微信 API | WeCom API v3 |
| 认证方式 | Bearer Token（API Key），同时支持 HTTP 和 gRPC |
